# COR24 Feedback Incorporation Plan

Feedback from Luther Johnson (MakerLisp / COR24 hardware designer), received
2026-03-13, regarding the Rust-to-COR24 translator's calling convention.

Source: `docs/research/20260314-cor24-feedback.txt`

## Summary of Feedback

Luther reviewed the translator's generated code (specifically the UART hello
example) and identified two areas where the translator deviates from the
intended COR24 calling convention:

### 1. Use `jal` for subroutine calls (not push/jmp)

**Current translator output (4 instructions, 10 bytes):**
```asm
    la      r2, .Lret_N       ; 4 bytes — compute return address
    push    r2                 ; 1 byte  — save return address
    la      r2, func          ; 4 bytes — load target
    jmp     (r2)              ; 1 byte  — jump
```

**Luther's recommended approach (3 instructions, 6 bytes):**
```asm
    push    r0                ; 1 byte  — save r0 (if needed as arg)
    la      r0, func          ; 4 bytes — load target address
    jal     r1, (r0)          ; 1 byte  — call: saves PC+1 in r1, jumps to (r0)
```

**Why:** `jal` is the COR24's dedicated call instruction. It saves the return
address in r1 automatically, eliminating the need to compute and push a return
label. The translator's current approach bypasses `jal` entirely, using manual
push/jmp sequences that are longer and don't match the hardware's intended
calling model.

**Impact:** Every non-tail function call saves 4 bytes (10→6) and 1 instruction.
For a program like recursive fibonacci with ~177 call instructions executed,
this is significant.

**Constraint:** `jal` always stores the return address in r1. The three
encodings are `jal r1,(r0)`, `jal r1,(r1)`, `jal r1,(r2)`. The translator
currently uses r2 as the call-scratch register; switching to `jal` means the
target address must be in r0, r1, or r2.

### 2. Standard subroutine prologue/epilogue

**Luther's recommended protocol:**
```asm
func:
    push    fp                ; save caller's frame pointer
    push    r2                ; save register variable
    push    r1                ; save return address
    mov     fp, sp            ; establish new frame

    ; ... function body ...

    pop     r1                ; restore return address
    pop     r2                ; restore register variable
    pop     fp                ; restore frame pointer
    jmp     (r1)              ; return
```

**Properties of this protocol:**
- r0: not saved — used for return value (and first argument on entry)
- r1: saved — holds return address from `jal`
- r2: saved — usable as a "register variable" across calls
- fp: saved — frame pointer for accessing local variables
- Arguments passed on the stack (not in registers)
- Result returned in r0

**Current translator approach:**
- Arguments in registers: r0=arg0, r1=arg1, r2=arg2 (MSP430 ABI mapping)
- Return address on stack (pushed before jmp, popped at ret)
- No standard prologue/epilogue — each function saves only what it uses

### 3. Arguments on stack (not in registers)

Luther's COR24 C compiler passes arguments on the stack and returns results
in r0. The translator currently maps MSP430's register-passing ABI (r12→r0,
r13→r1, r14→r2) directly.

**Luther's note:** "in general, we pass arguments on the stack, and get the
result back in r0."

## Implementation Plan

### Phase 1: Switch calls from push/jmp to jal (high value, low risk)

**Files:** `rust-to-cor24/src/msp430.rs`

**Change:** In `translate_call()`, replace the push-return-addr + la + jmp
sequence with:
```asm
    la      r0, func          ; load target (or use r1/r2 if r0 is live)
    jal     r1, (r0)          ; call — return address saved in r1 automatically
```

**Return sequence:** Change from `pop r2; jmp (r2)` to `jmp (r1)` since the
return address is now in r1 (not on the stack).

**Complications:**
- r0 may hold a live argument value at the call site. Need to push r0 first
  if it's live, or use r1/r2 for the target address.
- The translator currently uses r2 as the scratch register for call targets.
  With jal, valid target registers are r0, r1, r2 — any would work but
  register liveness must be considered.
- Tail call optimization: currently `la r2, func; jmp (r2)`. Could become
  `la r0, func; jmp (r0)` (no jal for tail calls since we don't need return).
- Recursive functions: the return address in r1 must be saved (pushed) before
  making a nested call, since the inner `jal` overwrites r1.

**Testing:** All 22 translator tests must pass. All 13 demos must produce
correct results. The reg-rs cor24 regression tests must be re-baselined.

**Savings estimate:** ~4 bytes per call site. The UART hello demo has 7 calls
→ saves ~28 bytes. Recursive fibonacci saves ~4 bytes per recursive call.

### Phase 2: Standard prologue/epilogue (medium value, medium risk)

**Files:** `rust-to-cor24/src/msp430.rs`

Add the standard `push fp; push r2; push r1; mov fp,sp` prologue and
matching epilogue to non-leaf functions. Leaf functions can skip saving
registers they don't clobber (Luther's optimization suggestion).

**Complications:**
- The translator currently handles frame pointer setup differently — it does
  `mov fp, sp` only in the reset vector prologue, not per-function.
- Spill slots are currently at fixed fp-relative offsets (r4→0(fp),
  r5→3(fp), etc.). Per-function fp setup would change these offsets.
- Need to distinguish leaf vs non-leaf functions to apply Luther's
  optimization: leaf routines can skip saving r1 if they don't call other
  functions.

**This is a larger refactor** that changes the spill register addressing model.
Should be done carefully with comprehensive testing.

### Phase 3: Stack-based argument passing (low priority)

**Files:** `rust-to-cor24/src/msp430.rs`

Change argument passing from registers (MSP430 ABI: r12→r0, r13→r1) to
stack-based (push args before call, callee reads from fp-relative offsets).

**Why low priority:** The current register-based approach works correctly and
matches what rustc generates. Changing to stack-based passing would require
remapping all argument references in every function, which is a fundamental
change to the translator. The main benefit would be compatibility with
Luther's C compiler output — which matters when mixing Rust and C code but
isn't needed for pure-Rust programs.

**Defer until:** There's a concrete need to link Rust-compiled and
C-compiled COR24 code.

## What to Update Now (documentation only)

Even before implementing the code changes, the docs should acknowledge
Luther's feedback and the intended calling convention:

1. **rust-to-cor24/README.md** — Add a "Calling Convention" section noting
   that the translator currently uses a simplified convention (register args,
   push/jmp calls) that differs from the standard COR24 C compiler convention
   (stack args, jal calls). Link to this plan.

2. **docs/rust-pipeline-demos.md** — Add a note that generated code uses
   push/jmp calls rather than jal, and that a future update will switch to
   jal per Luther's recommendation.
